## 3. Bugs to fix in the transformer

### 3.A `protein.recommendedName.ecNumber` is silently dropped â€” **HIGH PRIORITY**

**Evidence:**
- 19 occurrences in the 139-file corpus.
- `logging.warn` fires for every one (annotationType `protein.recommendedName.ecNumber`).
- The dispatch table `UniFireAnnotationTypeToSection.ts` has no entry for this key.
- `uniFireToUniProtkbConverter.ts` has explicit handlers for `keyword`, `protein.recommendedName.fullName`, `protein.alternativeName.fullName`, and `xref.GO`, but no `protein.recommendedName.ecNumber` branch â€” so it falls through to the dispatch table, fails the lookup, and gets warned + skipped.

**Target shape (from precomputed):**

```json
{
  "proteinDescription": {
    "recommendedName": {
      "fullName": { "value": "Cap-specific mRNA â€¦", "evidences": [...] },
      "ecNumbers": [
        { "value": "2.1.1.57",
          "evidences": [{ "evidenceCode": "ECO:0000256", "source": "ARBA", "id": "ARBA00011923" }] }
      ]
    }
  }
}
```

**Source UniFire shape:**

```json
{ "evidence": ["ARBA00011923","UR000001535"],
  "annotationType": "protein.recommendedName.ecNumber",
  "annotationValue": "2.1.1.57" }
```

**Type:** `ProteinNames.ecNumbers?: ValueWithEvidence[]` already exists in `src/uniprotkb/adapters/namesAndTaxonomyConverter.ts:17`.

**Required change in `uniFireToUniProtkbConverter.ts`:**

1. Hoist an `ecNumbers: ValueWithEvidence[] = []` accumulator next to `recommendedName` and `alternativeNames` (around line 76â€“77).
2. Add an explicit handler before the dispatch-table lookup (insert after the `protein.alternativeName.fullName` branch around line 116):

   ```ts

   if (annotationType === 'protein.recommendedName.ecNumber') {
     if (typeof annotationValue === 'string') {
       ecNumbers.push({ value: annotationValue, evidences });
     }
     continue;
   }
   ```
3. When assembling `proteinDescription` (currently lines 190â€“196), merge `ecNumbers` into `recommendedName`:

   ```ts
   let proteinDescription: ProteinNamesData | undefined;
   const recommendedNameWithEc = recommendedName || ecNumbers.length > 0
     ? {
         // fullName must exist on ProteinNames; if no recommendedName.fullName was found
         // but ecNumbers were, emit recommendedName with a synthetic empty-string fullName?
         // â†’ See open question below.
         ...(recommendedName ?? {}),
         ...(ecNumbers.length > 0 ? { ecNumbers } : undefined),
       }
     : undefined;
   if (recommendedNameWithEc || alternativeNames.length > 0) {
     proteinDescription = {
       ...(recommendedNameWithEc ? { recommendedName: recommendedNameWithEc as ProteinNames } : undefined),
       ...(alternativeNames.length > 0 ? { alternativeNames } : undefined),
   ```

**Open question for implementer:**

`ProteinNames.fullName` is non-optional in the type. If a UniFire entry has `protein.recommendedName.ecNumber` without a corresponding `protein.recommendedName.fullName`, two options:

- **(a)** Emit `recommendedName: { fullName: { value: '', evidences: [] }, ecNumbers }`. Type-safe, but renderer must tolerate empty fullName.
- **(b)** Drop the ecNumbers in that case (preserve existing behaviour for that subset) and log a warning.

Inspect the corpus for any accession where `recommendedName.ecNumber` exists without `recommendedName.fullName`:

Prefer (a) if any such entries exist; (b) is acceptable if none do.

**Tests to add** (in `__tests__/uniFireToUniProtkbConverter.spec.ts`):

### 3.B `comment.subcellular_location` emits free-text instead of structured

**Status:** Already documented in `spec.md` as a permanent thinness. Listed here for completeness.

**Evidence:**
- UniFire emits 632 `comment.subcellular_location` predictions in the corpus, each with `annotationValue` like `"Virion"` or `"Cell membrane; Single-pass type I membrane protein"`.

- `texts: [{value, evidences}]` (UniFire-derived)

**Possible future enhancement (not in scope here):** parse the free-text `"A; B; C"` form into one `subcellularLocations` array. Defer until the renderer work surfaces a need.

## 4. Dead / unreachable code

### 4.A `try/catch` inside the prediction loop has no reachable throw

Lines ~80â€“186 wrap each iteration in `try/catch`. Within the body, every operation is either a type-narrowing guard (`typeof === 'string'`), an object-literal construction, or an `Array.prototype.push`. None of these throw on the inputs `isValidUniFireModel` accepts (the validator already guarantees `evidence` is `string[]` and `annotationType` is `string`).

**Recommendation:** leave the `try/catch` in place. It is harmless and provides defence-in-depth if a future helper (e.g. an EC-number parser per Â§3.A or a cofactor parser per Â§3.C) starts to throw. Do **not** remove it as part of the EC-number fix; do **not** widen it either.

### 4.B `isValidUniFireModel` is duplicated logic vs. the structural type guard

The validator inspects `evidence` to be `string[]`. But `uniParcSubEntryConverter.ts:75â€“83` mutates the same `UniFireModel` in place, replacing each `evidence: string[]` with `Evidence[]` (typed objects) before that data ever reaches React. If the converter is later called on the *mutated* object, `isValidUniFireModel` will return `false` and the converter will `throw`.

**Action:** if/when wiring the converter into the component path, call it on the **original** (string-form) `evidence` array, *before* `uniParcSubEntryConverter` mutates it; or refactor `isValidUniFireModel` to accept both string-form and object-form evidence. Today neither is a bug because the converter has no caller in the component tree.

## 5. Evidence-source mapping is fragile

`constructPredictionEvidences` (in `uniParcSubEntryConverter.ts:39â€“54`) maps:

- ID starts with `ARBA` â†’ `source: 'ARBA'`
- otherwise â†’ `source: 'UniRule'`

In the empirical corpus (289 UniFire files, ~4,900 evidence references) only `ARBA*` and `UR*` IDs occur on UniFire inputs, so the current rule produces correct labels (`ARBA` or `UniRule`). However:

- If UniFire ever emits `RU*`, `PIRNR*`, or `PRU*` IDs (which appear in precomputed outputs but not in UniFire today), they would be mislabelled `UniRule`.
- The implicit assumption "non-`ARBA` â‡’ `UniRule`" is not documented in code.
- **Empirical asymmetry confirmed:** of the 250 precomputed files, **zero** contain an evidence ID starting with `UR`. Precomputed uses `RU*` â†’ `RuleBase` (1397Ã—), `ARBA*` â†’ `ARBA` (2470Ã—), `PIRNR*` â†’ `PIRNR` (77Ã—), `PRU*` â†’ `PROSITE-ProRule` (10Ã—). The `UR* â†’ UniRule` label is therefore a UniFire-internal convention with no corresponding precomputed evidence to validate it against; it should be treated as load-bearing display text owned by this code, not a passthrough.

**Recommendation (low priority):** turn the prefix mapping into an explicit table:

```ts
const SOURCE_BY_PREFIX: Array<[string, string]> = [
  ['ARBA', 'ARBA'],
  ['UR',   'UniRule'],
  // Add as needed: ['RU','RuleBase'], ['PIRNR','PIRNR'], ['PRU','PROSITE-ProRule']
];
const sourceFor = (id: string) =>
  SOURCE_BY_PREFIX.find(([p]) => id.startsWith(p))?.[1] ?? 'UniRule';
```

Defer until the corpus surfaces an actual mislabel â€” but flag it during code review.

## 6. Cross-references: transformer adds GO, precomputed has none

**Observation:** the gap analyser reports `Î” xrefs: -45` across 250 pairs â€” the transformer produces 45 `GO` cross-references (one per `xref.GO` UniFire prediction), but the precomputed corpus has **zero** `uniProtKBCrossReferences` of any database. The asymmetry persists at every corpus size (âˆ’28 at 50 pairs, âˆ’37 at 100, âˆ’45 at 250); precomputed never contributes xrefs.

**Verdict:** This is **not** a transformer bug. The original `spec.md` explicitly directs the transformer to map `xref.GO` to `uniProtKBCrossReferences[]`. The asymmetry is either:

- (a) the precomputed pipeline drops GO xrefs on its side (most likely);
- (b) precomputed never includes GO xrefs by design.

**Action:** keep current transformer behaviour. Confirm with the precomputed pipeline owner before relying on the assumption that "precomputed has zero xrefs" â€” that would inform whether the renderer needs a code path for empty `uniProtKBCrossReferences` at all when the data comes from precomputed, or only when it comes from UniFire. Not blocking this audit.

## 7. Comment count divergence is structural, not a bug

The per-pair detail shows the transformer routinely emits more comments than precomputed (e.g. `realComments=13` vs `precomputedComments=5` for `UPI0000000002-10245`). Root cause:

- UniFire emits one `comment.subcellular_location` prediction per location string, so a UniFire entry with 8 locations produces 8 separate `FreeTextComment` objects (`commentType: 'SUBCELLULAR LOCATION'`).
- Precomputed consolidates all subcellular locations into **one** `SubcellularLocationComment` with `subcellularLocations: [â€¦]` of length 8.

`Î” comments = âˆ’164` over 250 pairs lines up with the 632 vs 412 SUBCELLULAR LOCATION figure (`632 âˆ’ 412 = 220` consolidated, partially offset by other comment categories where precomputed exceeds the transformer in count). The relationship has been consistent across every corpus size sampled.

**Action:** **none in the transformer.** The renderer must group consecutive `SUBCELLULAR LOCATION` (and similarly for COFACTOR) comments for display, or the transformer can be enhanced later to consolidate them. Keep the simple "one prediction â†’ one comment" rule for now to avoid baking in fragile parsing.

## 7.5 Keyword content asymmetry â€” bidirectional (new at 250 pairs)

**New finding at the 250-pair corpus size.** At smaller corpus sizes the keyword delta was uniformly negative; at 250 pairs it flipped to `+6`. Investigation shows the direction is entry-specific:

| Direction | Pairs affected | Example | Extra keywords |
| :--- | ---: | :--- | :--- |
| precomputed > transformer | 5 | `UPI000000003C-111125` (8 vs 10) | `Isopeptide bond`, `Ubl conjugation` |
| transformer > precomputed | 1 | `UPI0000000004-10245` (6 vs 2) | `Host-virus interaction`, `Viral attachment to host cell`, `Viral envelope protein`, `Virus entry into host cell` |

Concrete reproductions:
- `transformer-gap/downloads/unifire/UPI000000003C-111125.json` has no `keyword` predictions for `Isopeptide bond` / `Ubl conjugation`, but `transformer-gap/downloads/precomputed/UPI000000003C-111125.json` lists them under `keywords[]`. Precomputed sourced those keywords from a rule path UniFire does not expose.
- `transformer-gap/downloads/unifire/UPI0000000004-10245.json` predicts four `keyword` annotations (host-virus interaction etc.) that precomputed strips.

**Verdict:** **not a transformer bug.** The transformer faithfully maps every `keyword` prediction it is given. Asymmetric content comes from upstream pipeline differences between UniFire and the precomputed processor:
- Some rule paths add keywords only at precomputation time (so UniFire-derived entries will lack them).
- Some UniFire predictions are filtered out by the precomputed processor (so UniFire-derived entries will *additionally* have them).

**Implication for the renderer:** keyword sets will differ between the two paths even for the same accession. The renderer cannot assume "UniFire-derived â‡’ subset of precomputed" â€” it must handle either path producing keywords the other does not.

**Action:** none in the transformer. Note in the renderer-side spec that keyword content is non-monotonic between the two sources.

**Optional follow-up (clearly out of this audit's scope, listed for completeness):** add a post-processing pass in the transformer that merges all `commentType: 'SUBCELLULAR LOCATION'` free-text comments into a single comment with a `texts: []` array of length N. This keeps the *shape* asymmetric with precomputed (free-text vs structured) but matches the *cardinality*, simplifying the renderer.
